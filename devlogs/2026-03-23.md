- `generate_slots(clinician_id, from_date, to_date, slot_duration_minutes=60)`
- Reads `ClinicianSchedule` rows; walks each calendar date in range
- Splits AM and PM windows into `slot_duration_minutes` (default 60) chunks
- Idempotent — skips slots that already exist (clinician + date + start_time)
- Bulk-inserts new `ClinicianTimeslot` rows with `status="available"`

### Appointment routes (`app/routes/appointment_routes.py`) — full rewrite

**Slot model corrections:**
- `POST /` (book): removed `slot.status = "booked"` — slots are no longer
  1:1 with appointments; booking never changes slot status
- `DELETE /` (cancel): removed `slot.status = "available"` — cancelling one
  appointment does not affect other patients on the same slot

**Status lifecycle enforced:**
```
pending → accepted | rejected | cancelled
accepted → reschedule_requested | cancelled
reschedule_requested → accepted (new slot) | cancelled
rejected / cancelled → terminal
```

**Reschedule flow:**
- Either party initiates `reschedule_requested`; `reschedule_reason` required
- C/S confirms by supplying `new_slot_id`; new slot must be `available`, same clinician
- Old slot is never touched

**Cancellation time gates:**
- > 48h: free, `cancellation_reason` required
- 24–48h: allowed with a warning in the response body
- < 24h patient: blocked (HTTP 403), directed to contact C/S
- < 24h C/S: blocked (HTTP 403), must use `reschedule_requested` flow

**`max_patients` auto-block:**
- On `pending → accepted`, counts accepted appointments for the slot
- If count ≥ `max_patients`, slot auto-blocks (`status = "blocked"`)

## Key design decisions made this session

| Decision | Detail |
|---|---|
| Slots are not 1:1 with appointments | A timeslot can hold multiple patients (clinician's internal N) |
| Slot status driven by C/S only | `available → blocked` via manual action or auto-block; booking/cancel never writes slot status |
| `"booked"` retired as a slot status | Valid slot statuses: `available`, `blocked` |
| `max_patients` opt-in limit | Nullable; auto-blocks slot when accepted count reaches limit |
| `reschedule_requested` status | Replaces `rescheduled`; either party initiates, C/S confirms with new slot |
| 24h hard block, 48h soft warning | Patient blocked <24h; C/S redirected to reschedule flow <24h |
| 60-minute slot default | Based on referential MMC observation (e.g. 14:00–15:00) |
| 60-day rolling generation | On schedule save; manual endpoint available for edge cases |

## Design rationale

### Slots are not 1:1 with appointments
The initial implementation incorrectly treated each timeslot as belonging to a
single patient. Observation of MMC's scheduling practice — where a clinician sees
multiple patients within the same hour block — required revising this. The slot
represents the clinician's availability window; multiple patients can be booked
into it up to an optional cap. This distinction is fundamental to the system's
accuracy as a model of real clinical workflow.

### Slot status written only by C/S
Separating slot status (availability) from appointment status (booking state)
reflects a deliberate division of control: the clinician or secretary owns the
schedule, and patients own their appointments. A patient booking or cancelling
should never silently alter the clinician's availability — that decision belongs
to the C/S role. This maps directly to the role separation identified in the
CeHRes user requirement analysis, where clinician autonomy over scheduling was
a stated requirement.

### `max_patients` as opt-in, nullable
Not all clinicians operate with a fixed patient cap per slot. Making `max_patients`
nullable means the system works for both cases without requiring clinicians to
configure a cap they may not want. When set, the auto-block behavior enforces the
cap without requiring manual intervention from C/S — reducing administrative load,
which was a primary process pain point identified during stakeholder discussion.

### Cancellation time gates (24h hard block, 48h soft warning)
Time-based cancellation rules reflect standard clinic policy at MMC and protect
both the clinician's schedule and other patients who may be waitlisted. The
asymmetry between patient and C/S rules (patient blocked at <24h, C/S redirected
to reschedule) acknowledges that C/S may have legitimate operational reasons to
act close to appointment time, but must do so through a flow that notifies and
re-accommodates the patient rather than simply cancelling.

### `reschedule_requested` as an explicit status
Using a dedicated status rather than cancelling and rebooking preserves the
appointment record and audit trail through the reschedule event. This supports
the thesis requirement for traceability of appointment state changes — a
committee or administrator can see that a reschedule was requested, by whom,
and what reason was given, rather than seeing a cancellation followed by a new
booking with no visible connection.

### Idempotent `generate_slots()`
Slot generation is safe to re-run without creating duplicates. This was a
deliberate choice to support the iterative development process — the function
can be called during testing, on schedule save, and on manual override without
requiring careful orchestration. Under CeHRes, system reliability during
formative evaluation cycles is important; idempotent operations reduce the risk
of data corruption during testing with real or simulated users.
