- `generate_slots()` is idempotent (add-only); if a clinician's schedule changes,
  previously generated future slots become orphaned

## Design rationale

### Auth skeleton without implementation
The decision to scaffold the auth layer (JWTManager, bcrypt helpers, route stubs
with documented contracts) without implementing the login logic reflects a
deliberate division of labor: a collaborator owns security decisions for this
project. Building the skeleton first means the collaborator inherits a clear
interface contract — documented request/response shapes, role claims, and deferred
`# TODO(security)` markers — rather than a blank file. This reduces the risk of
the auth implementation diverging from the rest of the system's conventions.

### JWT over session-based auth
`flask-jwt-extended` was selected over Flask's session mechanism because the
system is designed for eventual deployment behind a reverse proxy, with a
potential kiosk client on the same instance. Stateless JWT tokens work across
these surfaces without requiring shared session storage. The specific security
parameters (token lifetime, rotation, blocklist) are deferred to the
security-owning collaborator, who is better positioned to evaluate the tradeoffs
for the deployment environment.

### Stubs returning 501 (not 404 or 200)
Auth endpoints return HTTP 501 Not Implemented rather than a placeholder 200 or
a missing-route 404. This is intentional: 501 communicates to any consumer
(frontend, testing tool, collaborator) that the endpoint exists and has a defined
contract, but is not yet functional. It prevents silent failures where a caller
assumes a 200 means the login succeeded.

### Documenting design plans before implementing
The patient overlap check and schedule change handler were fully designed and
documented in this session rather than immediately implemented. This reflects the
CeHRes principle of explicit design review before development — particularly
important for logic that touches appointment integrity (overlap) and data
consistency (orphaned slots). Having the written plan in CLAUDE.md also means
any collaborator or future maintainer can understand the intended behavior before
reading the code.

### Touching boundaries allowed in overlap test
The overlap check uses `NOT (A.end ≤ B.start OR B.end ≤ A.start)`, which means
two appointments with touching boundaries (one ends at 14:00, another starts at
14:00) are not considered overlapping. This mirrors real-world scheduling practice
— a patient finishing a 13:00–14:00 appointment and starting a 14:00–15:00
appointment with a different clinician is a normal scenario, not a conflict.
