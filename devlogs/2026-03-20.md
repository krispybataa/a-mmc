## Design rationale

### App factory pattern (`create_app`)
Selected over a flat `app.py` structure to support environment-specific
configuration without coupling the app instance to a single config at import
time. This is the standard pattern for Flask applications that will run across
multiple environments (local dev, Docker, production) — a requirement given
that this project involves both a student development environment and a
collaborator-managed Docker deployment. Under the CeHRes Roadmap, this also
supports iterative development cycles where the system must remain runnable
at each stage without a full environment setup.

### VARCHAR over ENUM for status fields
PostgreSQL ENUMs require a schema migration to extend. Using VARCHAR preserves
the ability to add new statuses (e.g. a future "waitlisted" appointment state)
without a migration, which is appropriate for a system still under active design
during an undergraduate project timeline. CeHRes explicitly anticipates design
revision across formative cycles — flexible schema supports that.

### Child tables for HMO and Info (not JSONB arrays)
JSONB arrays would require deserializing and reserializing the entire array to
add or remove a single entry. Child tables keep each HMO accreditation and info
entry as a first-class row — addable, deletable, and queryable independently.
This also keeps query complexity low, which aligns with the project's
single-instance deployment model (no need to optimize for horizontal scale).

### Pre-generated timeslot table (`ClinicianTimeslot`)
Appointments reference a pre-generated slot row rather than storing raw datetime
ranges. This centralizes availability logic in one place and allows clinicians
and secretaries to manage availability explicitly (blocking slots, setting patient
caps) without the booking system having to infer availability from appointment
data. The tradeoff is that slots must be generated ahead of time, handled by a
60-day rolling window on schedule save.

### One Blueprint per domain entity
Routes are organized by domain entity rather than by role or feature. This keeps
each file's scope narrow and predictable — a convention that supports handoff to
collaborators who own specific domains (e.g. the auth collaborator only needs to
work in `auth_routes.py`).

### `.env.example` as a committed template
Secrets and credentials are never hardcoded or committed. `.env.example` serves
as a documented contract of what environment variables the system requires,
making onboarding new collaborators explicit rather than reliant on verbal
handoff — important for a multi-person undergraduate project.
