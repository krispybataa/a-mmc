# Alagang MMC — Handoff Prompt

## What this is
A digital health intervention for Makati Medical Center (MMC) designed to address
process hurdles in patient care coordination. Built as an undergraduate special
project at UP Manila under the CeHRes Roadmap (Human-Centered Design framework).
Stack: React + Vite frontend, Flask + SQLAlchemy backend, PostgreSQL 16.
All services Dockerized via compose.yaml in /a-mmc_infra.

Read CLAUDE.md in the project files — it is the source of truth for the entire
system. Everything below is operational context for this conversation.

## Operational rules — baked in from prior experience

### Rebuild pipeline
The developer's standard rebuild flow (run from a-mmc/ directory):
  docker build .\a-mmc_backend\ -t krispybata/a-mmc-backend:latest
  docker build .\a-mmc_frontend\ -t krispybata/a-mmc-frontend:latest
  cd .\a-mmc_infra\ ; docker compose up -d

Frontend-only changes only need the frontend build.
Backend-only changes only need the backend build.
Both needed when both layers are touched in the same mission.

Full reset (wipes DB volumes — requires re-seeding afterward):
  cd .\a-mmc_infra\ ; docker compose down -v
  then rebuild flow above

### When a rebuild is needed vs not
| Change type | Action needed |
|---|---|
| src/ JSX/JS/CSS files | Frontend rebuild |
| Backend Python files | Backend rebuild |
| New Python package in requirements.txt | Backend rebuild |
| New model added | Two-rebuild flow (see below) |
| Migration to apply | flask db upgrade in backend Exec only |
| compose.yaml changes | docker compose up -d |
| Nothing — just verifying | No rebuild |

### Two-rebuild flow for new models
Any mission that adds a new SQLAlchemy model requires this exact sequence:
1. Claude Code writes the model file locally
2. Rebuild backend (bakes new model code into image)
3. docker compose up -d
4. Developer runs in backend container Exec:
     flask db migrate -m "description"
5. Developer copies migration file out to local codebase:
     docker cp a-mmc-backend:/app/migrations/versions/<filename>.py \
       .\a-mmc_backend\migrations\versions\
   NOTE: Copy the specific file, not the directory, to avoid nesting issues
6. Rebuild backend again (bakes migration file into image)
7. docker compose up -d — flask db upgrade runs automatically on startup

Claude Code must NOT run flask db migrate inside the container during a mission.
Claude Code must NOT run docker exec commands to verify changes that require
a rebuilt image. State "rebuild required" in the report and stop.

### Curl testing rules
All curl smoke tests run in the backend container via Docker Desktop Exec,
hitting localhost:5000 directly — never through nginx. This bypasses proxy
routing issues. The developer runs these manually. Claude Code does not issue
curl commands during missions.

If the developer needs curl commands, I provide them as a separate block
outside the Claude Code mission prompt.

### Container Exec rule
If a command is best run via Docker Desktop Exec into a specific container,
I say so explicitly and the developer takes executive control. I do not
instruct Claude Code to run docker exec commands.

### API base URL convention
api.js baseURL is '/api'. All paths passed to api.get/post/patch/delete
must NOT include the /api prefix — they start with '/'.
Example: api.get('/clinicians/') not api.get('/api/clinicians/')
nginx proxies /api/* to the backend preserving the /api prefix.
Any new frontend code must follow this convention.

### Migration file location
Migration files live in:
  a-mmc_backend/migrations/versions/
Current migrations:
  59b8bcbc3cae_initial.py — initial schema (all 8 original tables)
  4da00536f0ca_add_admin_model.py — admin table
  <hash>_add_consultation_type.py — consultation_type on 3 tables
All three must be present in the image for flask db upgrade to work.

## Proposal constraints to keep in mind
- System scope: pre-consultation only (finding clinician, booking,
  managing appointment). No billing, diagnostics, post-consultation.
- Admin access to patient data: non-sensitive fields only (name,
  contact, email). No appointments, complaints, health data.
- The system operates independently of MMC's iHIMS and EMR.
- Primary demographic: 60+ users. Touch targets 44px min,
  base font 16px min throughout.
- Evaluation method: SRET (iterative) + UEQ (summative).
  System must be stable enough for real stakeholder testing.
